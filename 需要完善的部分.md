## 已完成 1：页面 bug 与 UI 优化

**原问题**：手动输入和 PDF 回填已经能跑通，但界面偏“字段堆叠”。用户不知道哪些字段来自发行安排公告、哪些来自招股书、哪些可由 Tushare 参考补充；预测结果区也需要更突出正式 T-6 口径和缺失字段风险。

**具体要做**：

- **表单分组**：手动输入已拆成「基础信息」「发行结构」「申购规则」「财务估值」几组，减少一次性长表单带来的误填。
- **PDF 回填状态**：发行安排公告 PDF、招股书 PDF 抽取后会展示已回填字段和来源。
- **字段来源标记**：对关键字段标明来源：询价公告 PDF、招股书、Tushare 行业 PE 参考、Tushare peer PE 参考。
- **缺失字段提示**：把模型可接受缺失和必须人工补充的字段区分开；招股书没抽到可比公司名单或 Tushare 匹配不到 PE 时，说明“模型可按缺失预测，但信息较少”。
- **预测结果层级**：沿用预测中签率、预测网下超额认购倍数、同板块历史分位、模型口径和数据截止日；SHAP 解释放在可展开区域。
- **口径防误用**：T-1/T+1 研究对照要弱化或明确标注，避免用户把询价后信息当成正式预测输入。

**当前状态**：已完成第一轮；后续如继续打磨，可再做更细的视觉布局和缺失字段分级。

## 已完成 2：Tushare 数据刷新运维化

**已具备**：

- `scripts/market_source.py` 可刷新全市场成交/收益、申万一级行业代码-名称映射、申万行业行情与 PE，并写入 CSV/SQLite。
- `scripts/peer_valuation.py` 可按申万一级行业成分和 `daily_basic.pe_ttm` 计算同行业 peer PE 均值/中位数。

**已补充**：

- `sw_daily` 等接口存在频率限制；现在刷新为空或失败时会保留既有 `sw_level1_market_daily_tushare.csv` 缓存，避免一次失败中断全部流程或覆盖有效数据。
- 刷新脚本产出的短期行情缓存默认作为运行数据，不要求每次提交仓库；需要固定演示数据时再单独提交。
- Streamlit Cloud 需要在 Secrets 配置 `TUSHARE_TOKEN`；无 token 时行业 PE / peer PE 回填不可用，但手动输入和 PDF 抽取仍可用。

**不做**：AkShare 备源。

## 已完成 3：Tushare 行业 PE / peer PE 接入页面

**边界**：

- `industry_pe_at_ipo` 是 T-6 市场行业 PE，可以用 Tushare 申万行业 PE 做近似参考。
- `comparable_pe_avg_ex_nonrecurring` 当前口径改为：招股书披露可比公司名单 + Tushare `daily_basic.pe_ttm` 计算 peer PE 均值；发行公告/投资风险特别公告的 T-1 估值比较不作为来源。

**具体要做**：

- 回填后仍保留手动输入框，允许用户覆盖。

## 4. 历史相似新股检索

**目标**：在预测结果下方展示 5-10 只相似历史新股，帮助用户判断预测是否合理。

**候选相似维度**：

- 板块、申万一级行业、发行规模、网下发行比例、战略配售比例、预计募资额、营收规模、行业 PE、同板块历史热度。

**注意**：相似样本展示只能用于解释，不能引入未来标签作为新股预测输入。

## 5. 策略层评估

**目标**：从“模型误差”进一步走到“申购决策有没有用”。

**候选评估**：

- 按预测中签率分位分组，观察真实中签率是否单调。
- 按预测中签率排序，模拟优先申购池。
- 结合首日涨幅，做简单收益/风险分层。

**注意**：收益模拟要明确假设，不要把样本内结果写成真实可交易收益。
